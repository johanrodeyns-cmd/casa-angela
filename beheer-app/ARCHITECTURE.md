# ARCHITECTURE — Casa Angela Beheer-app

> Auto-ingeladen via `@ARCHITECTURE.md` in `CLAUDE.md`. Houd deze kaart actueel bij elke structurele wijziging (nieuw bestand, nieuwe/verplaatste functie, gewijzigde laadvolgorde, nieuw Firestore-pad of -veld, nieuwe dependency, nieuwe Cloud Function). Zie de architectuur-verificatieregel in de globale CLAUDE.md.

Vanilla-JS single-page PWA op Firebase Hosting (geen build/bundler): prijskalender, boekingenbeheer, beschikbaarheidskalender, aankomst/vertrek-checklists en nutsvoorzieningen ("Nuts": zonnestroom, netstroom, water, gas) voor vakantiehuis Casa Angela. **Eigen** Firebase-project `casa-angela-jr` op het **Blaze-plan** (nodig voor uitgaande netwerk-requests vanuit Cloud Functions), volledig los van de andere projecten. Gedeelde data tussen twee gebruikers (Johan + Tinneke) i.p.v. per-uid isolatie. iCal-sync met Airbnb/Booking.com en APsystems-zonnepaneeldata (met storingsmonitor) elk in eigen Cloud Functions.

## Datamodel (Firestore — gedeeld, geen per-uid pad)

| Pad | Vorm |
|---|---|
| `pricing/{YYYY-MM-DD}` | `{ date, airbnbPrice }` — enkel de Airbnb-basisprijs wordt opgeslagen |
| `settings/pricingFormula` | `{ booking: {factor, offset}, direct: {factor, offset}, friends: {factor, offset} }` — afgeleide prijs = `airbnbPrice * factor + offset`, per type instelbaar via een instellingenscherm |
| `bookings/{id}` | `{ dateFrom, dateTo, nights, name, language, phone, adultsCount, childrenCount, remark, platform, price, createdBy, createdAt }` — geen aparte statusveld; annuleren = document verwijderen |
| `syncedBlocks/{id}` | `{ source: 'airbnb'|'booking', dateFrom, dateTo, icalUid }` — automatisch gesynchroniseerde bezette periodes zonder gastgegevens, apart van `bookings` |
| `icalFeeds/{airbnb|booking}` | `{ url, lastSyncedAt }` |
| `checklists/{aankomst|vertrek}` | `{ items: [{ id, text, checked, order }] }` |
| `auditLog/{autoId}` | `{ collection, docId, action: 'create'\|'update'\|'delete', email, timestamp: serverTimestamp(), before, after }` (US-5.1) — append-only: `create` toegestaan (enkel onder het eigen, servergevalideerde e-mailadres), `update`/`delete` altijd geweigerd, ook voor de eigenaar (`firestore.rules`) |
| `meta/allowedUsers` | `{ emails: ['johan.rodeyns@gmail.com', 'tinbogaerts@gmail.com'] }` (spiegel van de Firestore-rule, voor UI-weergave "ingelogd als") — **nog niet aangemaakt**, whitelist zit voorlopig hardcoded in `firestore.rules` + `index.html` (US-0.2) |
| `settings/apsystems` | `{ appId, appSecret, sid, eid, alertEnabled, alertEmail, monitorState: {date, lastToday, consecutiveFlat, alerted, lastCheckAt} }` — gedeeld tussen Johan en Tinneke (één zonnepaneelsysteem), geen per-uid pad. `appSecret` staat in Firestore (niet Secret Manager) omdat hij per API-call client-side wordt meegegeven aan de Cloud Function — enkel toegankelijk voor de 2 whitelisted e-mailadressen via `firestore.rules` |
| `cache/apsystems` | `{ [key]: { data, fetchedAt } }` — 30 min TTL-cache voor de resterende live APsystems-calls (`summary`, `today_{date}`, `meter_energy_hourly_{date}`); voorkomt herhaalde calls tegen de 1000/maand-quota bij elk tab-bezoek. Sinds v0.46.0 enkel nog gebruikt voor "vandaag"-data (intraday, nooit gearchiveerd) — de Dag/Maand/Jaar-historie gebruikt deze cache niet meer, die leest rechtstreeks uit `energyHistory` |
| `waterReadings/{id}` | `{ date, reading, createdBy, createdAt }` (v0.35.0) — manuele meterstand (m³) voor de Water-sub-tab, één document per meting |
| `gasReadings/{id}` | `{ date, reading, createdBy, createdAt }` (v0.35.0) — idem voor Gas |
| `energyHistory/{YYYY-MM}` | `{ solar: number[], grid: number[], daysInMonth, complete, throughDate, updatedAt }` (US-6.11) — permanent dagverbruik (kWh, index 0 = dag 1) per kalendermaand, geschreven door `casaAngelaArchiveEnergy`; `complete: true` zodra de maand volledig voorbij is (wordt daarna nooit meer herschreven); `throughDate` (`'YYYY-MM-DD'`\|`null`, sinds v0.47.1) = laatste dag waarvoor dit document effectief data bevat. Client-`read`-only, geen client-`write` (zie `firestore.rules`). Sinds v0.46.0 ook de rechtstreekse databron voor Zonnestroom/Netstroom's Dag/Maand/Jaar-historie (`loadEnergyHistory()` in `index.html`, geen live APsystems-call meer) |
| `energyHistory/_meta` | `{ earliestAvailableMonth: 'YYYY-MM' }` (US-6.11) — de vroegst gekende maand met APsystems-data (grens vóór de meter-installatie), zodat de terugwaartse archivering die grens niet elke dag opnieuw hoeft te bevragen |

## Cloud Functions (`functions/`, Node 22, regio `us-central1`)

- `syncIcalFeeds` (onSchedule, elke 3u) — haalt beide iCal-feeds op, parsed, vervangt `syncedBlocks` per bron volledig; logt `auditLog`-records (via `diffSyncedBlocks`, `email: 'systeem (automatische sync)'` — er is geen `request.auth` bij een schedule-trigger).
- `syncIcalFeedsNow` (onCall) — zelfde logica, handmatig getriggerd vanuit de "🔄 Nu synchroniseren"-knop; controleert zelf `request.auth.token.email` tegen de whitelist via `requireAllowedUser()` (Admin SDK omzeilt Firestore rules, dus deze check gebeurt in de functie zelf) — `auditLog`-records krijgen hier het echte e-mailadres van de gebruiker die op de knop klikte.
- `syncSource(source, actorEmail)`/`syncAllSources(actorEmail)` — gedeelde sync-logica; `diffSyncedBlocks(before, after)` (**eigen kopie**, zelfde reden als `parseIcalEvents`/`mergeSyncedBlocks` — zie hieronder) vergelijkt de volledige oude/nieuwe blokkenlijst per bron op id en logt enkel een `create`/`update`/`delete` voor blokken die écht veranderd zijn; zonder deze diff zou elke 3u-sync alle (ongewijzigde) blokken opnieuw als delete+create loggen, want `mergeSyncedBlocks` is altijd een volledige vervanging, geen incrementele diff.
- `casaAngelaSummary`/`casaAngelaInverters`/`casaAngelaToday`/`casaAngelaEnergy` (onCall, v0.34.0) — CORS-proxy naar `api.apsystemsema.com:9282` (die stuurt geen `Access-Control-Allow-Origin`, dus directe browser-calls worden geblokkeerd); elke call ondertekent zelf via `buildHeaders()` (App ID/Secret/SID komen uit `request.data`, niet uit Firestore server-side — de client leest/schrijft `settings/apsystems` zelf en geeft de waarden door per call).
- `casaAngelaMeterEnergy` (onCall, v0.37.0, `hourly` erbij in v0.40.0) — Netstroom (`exported`), Meter-level Data API (`/user/api/v2/systems/{sid}/devices/meter/period/{eid}`, level hourly/daily/monthly/yearly — was tot v0.47.2 foutief `/installer/api/v2/...`; dat namespace bestaat wel in de officiële "APsystems OpenAPI User Manual" maar uitsluitend voor de storage/battery-endpoints, niet voor meter/period, zie US-6.11-vervolgnotitie in `user-stories.md`). Zelfde ondertekening/credentials als de andere Nuts-calls hierboven — geen apart auth-mechanisme nodig, enkel een meter-eid i.p.v. een inverter/ECU-eid. Vervangt een eerdere, inmiddels afgebouwde aanpak die via een EMA-website-sessiecookie moest scrapen omdat deze officiële Meter-level API toen nog niet gekend was. `energy_level=minutely` bestaat ook volgens de documentatie, maar gaf HTTP 500 voor deze meter (zelfde fout-namespace-oorzaak, achteraf pas herkend) — de `casaAngelaMeterToday`-functie die daarvoor gebouwd was, is in v0.40.0 alweer verwijderd.
- `casaAngelaMonitor` (onSchedule `0 9-18 * * *`, `Europe/Madrid`, secret `GMAIL_APP_PASSWORD`) — elk uur (9-18u Spaanse tijd) haalt `processNutsMonitor()` de dagopbrengst op en vergelijkt die via `decideAlertState` (2× op rij vlak → storingsmail; nadien stijging → herstelmail; state bewaard in `settings/apsystems.monitorState`). `casaAngelaMonitorRunNow` (onCall) forceert één check handmatig ("Forceer check"-knop); `casaAngelaMonitorTest` (onCall) stuurt een losse testmail, los van de monitor-state.

**Bewust geen `node-ical`/`axios`**: die combinatie bleek bij het laden onherstelbaar vast te hangen op deze ontwikkelmachine (project staat op een NAS-gekoppelde netwerkschijf; `node-ical` require't `axios` onvoorwaardelijk, en die hing zowel via CJS `require` als ESM `import`, ongeacht axios-versie). Dat blokkeerde `firebase deploy`'s lokale "analyseer source code"-stap (vaste 10s-timeout, te overschrijven met env var `FUNCTIONS_DISCOVERY_TIMEOUT=<seconden>` — nuttig als toekomstige deploys op deze machine traag blijven door de netwerkschijf). Oplossing: eigen minimale VEVENT-parser (`parseIcalEvents`, zie `logic.js`) + Node's ingebouwde `fetch()`, zonder externe iCal-dependency. Geen secrets nodig voor de iCal-sync (URL's van Airbnb/Booking.com zijn publieke, niet-geïndexeerde links die de gebruiker zelf aanlevert via het instellingenscherm) — de Nuts-monitor heeft wél een secret nodig, zie hierboven.

## Externe dependencies

Client: Firebase SDK (modular: Auth, Firestore, **Functions** — `httpsCallable` voor `syncIcalFeedsNow` + de Nuts-callables) via CDN; Chart.js 4.4.1 sinds v0.34.0, lazy geladen via dynamische `import()` van de jsdelivr `+esm`-build (`loadNutsChartJs()`) — enkel voor de Nuts-tab, geen impact op de rest van de app. PWA via `manifest.json` (geen service worker, zoals Gezondheid) — installeerbaar op GSM-startscherm. Server: `functions/` heeft een eigen `package.json`/`node_modules` (`firebase-admin`, `firebase-functions`, `nodemailer` sinds v0.34.0) — enige plek in dit project met npm-dependencies. Externe dienst: `api.apsystemsema.com` (zonnepaneeldata, HMAC-signed, geen SDK).
