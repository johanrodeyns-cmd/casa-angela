# ARCHITECTURE — Casa Angela Beheer-app

> Auto-ingeladen via `@ARCHITECTURE.md` in `CLAUDE.md`. Houd deze kaart actueel bij elke structurele wijziging (nieuw bestand, nieuwe/verplaatste functie, gewijzigde laadvolgorde, nieuw Firestore-pad of -veld, nieuwe dependency, nieuwe Cloud Function). Zie de architectuur-verificatieregel in de globale CLAUDE.md.

Vanilla-JS single-page PWA op Firebase Hosting (geen build/bundler): prijskalender, boekingenbeheer, beschikbaarheidskalender, aankomst/vertrek-checklists en nutsvoorzieningen ("Nuts": zonnestroom, netstroom, water, gas) voor vakantiehuis Casa Angela. **Eigen** Firebase-project `casa-angela-jr` op het **Blaze-plan** (nodig voor uitgaande netwerk-requests vanuit Cloud Functions), volledig los van de andere projecten. Gedeelde data tussen twee gebruikers (Johan + Tinneke) i.p.v. per-uid isolatie. iCal-sync met Airbnb/Booking.com en APsystems-zonnepaneeldata (met storingsmonitor) elk in eigen Cloud Functions.

## Cloud Functions (`functions/`, Node 22, regio `us-central1`)

- `syncIcalFeeds` (onSchedule, elke 3u) — haalt beide iCal-feeds op, parsed, vervangt `syncedBlocks` per bron volledig; logt `auditLog`-records (via `diffSyncedBlocks`, `email: 'systeem (automatische sync)'` — er is geen `request.auth` bij een schedule-trigger).
- `syncIcalFeedsNow` (onCall) — zelfde logica, handmatig getriggerd vanuit de "🔄 Nu synchroniseren"-knop; controleert zelf `request.auth.token.email` tegen de whitelist via `requireAllowedUser()` (Admin SDK omzeilt Firestore rules, dus deze check gebeurt in de functie zelf) — `auditLog`-records krijgen hier het echte e-mailadres van de gebruiker die op de knop klikte.
- `syncSource(source, actorEmail)`/`syncAllSources(actorEmail)` — gedeelde sync-logica; `diffSyncedBlocks(before, after)` (**eigen kopie**, zelfde reden als `parseIcalEvents`/`mergeSyncedBlocks` — zie hieronder) vergelijkt de volledige oude/nieuwe blokkenlijst per bron op id en logt enkel een `create`/`update`/`delete` voor blokken die écht veranderd zijn; zonder deze diff zou elke 3u-sync alle (ongewijzigde) blokken opnieuw als delete+create loggen, want `mergeSyncedBlocks` is altijd een volledige vervanging, geen incrementele diff.
- `casaAngelaSummary`/`casaAngelaInverters`/`casaAngelaToday`/`casaAngelaEnergy` (onCall, v0.34.0) — CORS-proxy naar `api.apsystemsema.com:9282` (die stuurt geen `Access-Control-Allow-Origin`, dus directe browser-calls worden geblokkeerd); elke call ondertekent zelf via `buildHeaders()` (App ID/Secret/SID komen uit `request.data`, niet uit Firestore server-side — de client leest/schrijft `settings/apsystems` zelf en geeft de waarden door per call).
- `casaAngelaMonitor` (onSchedule `0 9-18 * * *`, `Europe/Madrid`, secret `GMAIL_APP_PASSWORD`) — elk uur (9-18u Spaanse tijd) haalt `processNutsMonitor()` de dagopbrengst op en vergelijkt die via `decideAlertState` (2× op rij vlak → storingsmail; nadien stijging → herstelmail; state bewaard in `settings/apsystems.monitorState`). `casaAngelaMonitorRunNow` (onCall) forceert één check handmatig ("Forceer check"-knop); `casaAngelaMonitorTest` (onCall) stuurt een losse testmail, los van de monitor-state.

**Bewust geen `node-ical`/`axios`**: die combinatie bleek bij het laden onherstelbaar vast te hangen op deze ontwikkelmachine (project staat op een NAS-gekoppelde netwerkschijf; `node-ical` require't `axios` onvoorwaardelijk, en die hing zowel via CJS `require` als ESM `import`, ongeacht axios-versie). Dat blokkeerde `firebase deploy`'s lokale "analyseer source code"-stap (vaste 10s-timeout, te overschrijven met env var `FUNCTIONS_DISCOVERY_TIMEOUT=<seconden>` — nuttig als toekomstige deploys op deze machine traag blijven door de netwerkschijf). Oplossing: eigen minimale VEVENT-parser (`parseIcalEvents`, zie `logic.js`) + Node's ingebouwde `fetch()`, zonder externe iCal-dependency. Geen secrets nodig voor de iCal-sync (URL's van Airbnb/Booking.com zijn publieke, niet-geïndexeerde links die de gebruiker zelf aanlevert via het instellingenscherm) — de Nuts-monitor heeft wél een secret nodig, zie hierboven.
